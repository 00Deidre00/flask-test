1) Problem Statement Canvas

고객 (Customer)

서울시 공공자전거 재배치 담당자 및 관리자.

컨텍스트 (Context)

실시간으로 변화하는 대여소별 자전거 수요와 공급 상황에 대응하며 재배치 업무를 수행하는 환경.

근본 문제 (Root Problem)

재배치 담당자의 임의적이고 자체적인 판단에 의존하는 현재 운영방식은 비효율을 초래하고 기회비용을 증가시킴.

감정적 영향 (Negative Emotions)

과도한 피로도, 불확실성에서 오는 스트레스.

정량적 영향 (Negative Outcomes)

기회비용 증가, 낮은 재고 만족률, 사용자의 대여/반납 불가 상황 빈번 발생.

대안 (Current Alternatives)

실시간 거치율만 확인하며 담당자의 경험과 직감에 의존하는 기존 운영방식.

대안의 단점 (Downsides of Alternatives)

미래 수요 예측 불가, 전역 최적 경로 탐색 불가능, 주관적 판단으로 인한 성과 편차 발생.

2) 솔루션 (Solution in 3 Lines)

  • Line 1: Random Forest 예측 모델을 통해 시간대별 대여/반납 수요를 예측하고, 이를 기반으로 대여소별 '적정 재고 수준(Bound)'을 동적으로 설정합니다.
  • Line 2: 클러스터링, Tabu Search, MIP 수리모형을 결합한 3단계 최적화 알고리즘으로 담당자별 최적 재배치 경로와 수량을 도출합니다.
  • Line 3: 관리자와 재배치 담당자를 위한 전용 애플리케이션(UBD Application)을 통해 이 모든 과정을 시각적으로 제공하고 업무 효율을 극대화합니다.

3) 기능 명세 (Feature List)

공통 기능

  • 실시간 재고 현황 시각화: 부족(빨강), 적정(회색), 초과(파랑) 색상으로 표시
  • 상세 정보 조회: 마커 선택 시 '적정 재고 수준' 상/하한선 그래프 제공

관리자 화면

  • 알고리즘 실행 기능 (담당자 수 입력)
  • 군집화 결과 확인 (지도에서 시각적으로 확인)
  • 결과 전송 (담당자에게 최적 경로 및 작업량 전송)

재배치 담당자 화면

  • 최적 경로 확인 (총 예상 소요 시간 및 거리)
  • 순차적 작업 안내 (다음/다다음 경로 표시)
  • 진행률 시각화 (작업 완료율, 시간 경과율 프로그레스 바)
  • 작업량 안내 (정확한 재배치 수량 제시)

4) 시스템 아키텍처 및 흐름

요청(App) → 데이터 전송(API) → 연산(Algorithm) → 결과 전송(API) → 시각화(App)

사용자의 요청에 따라 실시간 API 데이터를 Flask 서버로 전송하고, 서버에서 알고리즘 연산 후 그 결과를 다시 앱으로 보내 지도 위에 시각화하는 구조로 동작합니다.

5) 와이어 프레임 (스케치)

아래는 제안하는 애플리케이션의 주요 화면 흐름을 나타낸 와이어프레임입니다. 사용자는 시작 화면(Screen 1)에서 로그인을 거쳐(Screen 2) 메인 지도 화면으로 진입합니다(Screen 3). 메인 화면에서는 추천 경로, 작업 진행률을 확인할 수 있으며, 각 대여소 마커를 클릭하면 상세 정보(Screen 3-1)를 볼 수 있습니다. 모든 작업을 완료하면 완료 메시지(Screen 3-2)가 표시됩니다.
